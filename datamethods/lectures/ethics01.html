Chapter 10: Data Quality and Inference Errors

The Challenge of Big Data Errors

The Problem:

  • Big data brings new opportunities but also new error risks
  • Data often “repurposed” from original collection intent
  • Errors can be variable (random) or systematic (directional)

Key Examples:

  • Literary Digest (1936): Bad sampling led to wrong election prediction
  • Google Flu Trends (2012-2013): Overestimated flu incidence by nearly double

Why This Matters:

  • Without proper “data hygiene,” errors compound quickly
  • New data types are so large/complex that traditional error detection fails
  • Need systematic frameworks to understand and mitigate errors

The Total Error Framework

Traditional Survey Error Model Extended to Big Data:

\[ \text{Total Error} = \text{Row Error} + \text{Column Error} + \text{Cell Error} \]

Row Errors:

  • Omissions: Missing population elements
  • Duplications: Same entity multiple rows
  • Erroneous inclusions: Out-of-scope elements

Column Errors:

  • Specification errors: Mislabeled variables
  • Metadata errors: Incorrect column definitions

Cell Errors:

  • Content errors: Wrong values
  • Specification errors: Definition mismatches
  • Missing data: Empty cells

Row Errors: Coverage and Selection Issues

Omissions (Coverage Bias):

  • Formula: \(B_{NC} = (1 - N/N_T)(\bar{V}_C - \bar{V}_{NC})\)
  • Where \(N\) = observed, \(N_T\) = target population, \(\bar{V}_C\) = mean value in covered, \(\bar{V}_{NC}\) = mean in non-covered
  • Example: Google searches exclude non-Internet users

Duplications:

  • Same person multiple tweets/searches
  • Can bias individual-level vs. activity-level inferences
  • Mitigation: Deduplication or weighting by \(1/d\)

Erroneous Inclusions:

  • Bots, automated systems, out-of-scope records
  • Example: Computer-generated tweets vs. human tweets

Google Flu Trends: A Case Study

What Happened:

  • 2009-2011: Remarkably accurate flu tracking
  • 2012-2013: Overestimated CDC rates by nearly 100%
  • Failed for 100 out of 108 weeks starting July 2012

Root Causes (Lazer et al.):

  • Big Data Hubris: Volume doesn’t compensate for methodological flaws
  • Algorithm Dynamics: Google’s search suggestions created feedback loops
  • Blue Team Dynamics: Platform changes broke predictive relationships

Key Lesson:

Successful big data analyses today may fail tomorrow without robust methodological foundations

Analysis Errors: Even with Perfect Data

Three Major Issues (Fan et al.):

  1. Noise Accumulation: Adding uninformative variables overwhelms signal

    • With 1,000 variables, only first 10 informative
    • Classification degrades when \(m > 200\) variables included

  2. Spurious Correlations: High dimensionality creates false discoveries

    • With 800 independent variables: 50% chance of |correlation| > 0.4

  3. Incidental Endogeneity: Covariates correlated with residuals by chance

    • Leads to biased model selection and predictions

Error Models for Continuous Variables

Basic Error Model: \[y_{rc} = \mu_{rc} + \varepsilon_{rc}\]

Two Types of Errors:

Variable (Random) Errors:

  • \(\text{Cov}(y_r, y_s) = 0\) for \(r \neq s\)
  • Examples: Keying errors, measurement noise

Correlated (Systematic) Errors:

  • \(\varepsilon_{rj} = b_j + e_{rj}\) where \(b_j\) is source-specific bias
  • Examples: Calibration differences across sensors

Impact on Correlation Analysis

Key Formula: \[\rho_{y|cd} = \sqrt{R_c R_d} \rho_{\mu|cd} + \sqrt{\rho_c \rho_d}\]

Where:

  • \(R_c\) = reliability ratio (signal vs. total variance)
  • \(\rho_c\) = intra-source correlation (systematic error effect)

Variable Errors Only: Attenuate correlations by \(\sqrt{R_c R_d}\)

Systematic Errors: Can both inflate and attenuate unpredictably

Aggregation Helps: Combining \(n\) observations reduces variable error impact - Variable error variance: \(O(n^{-1})\) - But systematic errors persist regardless of \(n\)

Rare Population Analysis Challenges

The Terrorist Detector Example: - 99.9% accurate classifier - 1 in 1,000,000 true terrorists - Person classified as terrorist: Only 0.1% chance of being correct!

Why So Wrong? \[\text{PPV} = \frac{(1-\theta)\pi}{\pi(1-\theta) + (1-\pi)\phi} = \frac{0.999 \times 10^{-6}}{0.999 \times 10^{-6} + 0.999999 \times 0.001} \approx 0.001\]

General Principle: For very rare events, even highly accurate classifiers yield mostly false positives

Chapter 11: Bias and Fairness

The Challenge of Algorithmic Fairness

Core Problem: - No universal definition of “fair” - Multiple competing metrics that cannot all be optimized simultaneously - Must start with social and policy goals, then map to statistical properties

Key Insight: Machine learning models cannot be separated from social and ethical context - accuracy and fairness must both be considered

Sources of Bias in ML Pipeline

1. Sample Bias - Training data â‰ application population - Systematic exclusions (e.g., bail data only from released individuals) - Over-policing effects in criminal justice data

2. Label (Outcome) Bias
- Using arrests vs. crimes committed - Measurement error varies across groups - Differential accuracy in data collection

3. Pipeline Bias - Data ingestion: Record matching accuracy varies by group - Feature engineering: Geocoding misses homeless populations
- Modeling: Performance varies across groups

4. Application Bias - Intervention effectiveness varies by group - Discriminatory application of “fair” models

Fairness Metrics: Confusion Matrix Foundation

Key Ratios from Confusion Matrix:

  • False Positive Rate (FPR): \(\frac{FP}{FP + TN}\)
  • False Negative Rate (FNR): \(\frac{FN}{FN + TP}\)
  • Precision (PPV): \(\frac{TP}{TP + FP}\)
  • Recall (Sensitivity): \(\frac{TP}{TP + FN}\)
  • False Discovery Rate (FDR): \(\frac{FP}{FP + TP}\)
  • False Omission Rate (FOR): \(\frac{FN}{FN + TN}\)
  • Specificity: \(\frac{TN}{TN + FP}\)

Critical Point: These metrics focus on different populations (actual positives vs. predicted positives vs. total population)

Punitive vs. Assistive Applications

Punitive (e.g., Bail Denial) Focus on False Positives

  • Count of FP: Equal numbers wrongly detained
  • Group-adjusted FP: Equal rates by group size
  • FDR: Among detained, equal error rates
  • FPR: Among innocent, equal error rates

Assistive (e.g., Food Benefits) Focus on False Negatives

  • Count of FN: Equal numbers missed
  • Group-adjusted FN: Equal miss rates by group size
  • FOR: Among not served, equal need rates
  • FNR: Among needy, equal miss rates

Trade-off Reality: When prevalence differs across groups, you cannot eliminate disparities in both group-adjusted errors and condition-specific rates simultaneously

The Fairness Tree Decision Framework

Key Decision Points:

  1. Is your intervention punitive or assistive?
    • Punitive â†’ Focus on false positives
    • Assistive â†’ Focus on false negatives
  2. What type of harm matters most?
    • Individual harm from inclusion/exclusion errors
    • Societal harm from disparate treatment
  3. Are resources constrained?
    • Constrained â†’ Consider recall-based metrics
    • Unconstrained â†’ Traditional error rates applicable

Resource-Constrained Programs

Special Case: When program serves small fraction of those with need

Problem: Traditional metrics become uninformative - FOR approaches overall prevalence - FNR approaches 1.0

Solution: Focus on Recall (sensitivity) \[P(\text{received benefit} | \text{need}, \text{group}_i) = C\]

Alternative Approaches: - Equal recall across groups: \(\frac{\text{recall}_i}{\text{recall}_j} = 1\) - Recall proportional to need: \(\frac{\text{recall}_i}{\text{recall}_j} = \frac{p_i}{p_j}\)

Mitigating Bias: Practical Approaches

1. Auditing Existing Systems

  • Measure bias in current (non-ML) processes
  • Compare ML improvements to status quo
  • Tools like Aequitas for systematic assessment

2. Model Selection Integration

  • Consider equity alongside performance metrics
  • Create “menu” of options showing trade-offs
  • Example: Precision 0.75 + FDR ratio 1.3 vs. Precision 0.64 + FDR parity

3. Deployment Strategies

  • Different thresholds by group to achieve equity
  • Expand program size to avoid denying interventions
  • Random selection from expanded pool (e.g., top 1000 for 150 slots)

Chapter 12: Privacy and Confidentiality

The Fundamental Challenge

Core Tension:

Research value requires data access for replication, validation, and extension BUT more access increases confidentiality risk

Traditional Safeguards Now Obsolete:

  • Anonymization: Nearly impossible with rich big data
  • Informed consent: Too complex for comprehensible consent

New Reality:

Multiple datasets linked by various actors beyond traditional researchers, with no clear data stewards

Key Definitions

Utility: Value from data use

  • Private sector: Amazon, Google profit model
  • Research: Answering policy questions

Privacy: Appropriate flow of information

  • Context-dependent sharing decisions
  • Not absolute secrecy

Confidentiality: Preserving authorized access restrictions

  • Technical/legal protection measures

Risk: Probability of reidentification

  • Increases with data richness and external linking

Harm: Negative consequences from disclosure

  • Type of information matters more than fact of disclosure

The Utility-Privacy Trade-off

Fundamental Relationship:

  • More data access â†’ Higher utility â†’ Higher risk
  • External data growth shifts frontier inward
  • Organizations must balance competing demands

As external data increases, achieving same utility U requires reducing privacy from Pâ‚ to Pâ‚‚*

Historical Solution: Statistical agencies managed this trade-off with:

  • Legal mandate and capacity
  • Understanding of user community
  • Technical expertise

Why Big Data Breaks Traditional Models

Reidentification Examples:

  • Governor Weld (1990s): Medical records + voter files
  • AOL searches (2006): Browsing patterns revealed identity
  • Target teenager (2012): Shopping patterns revealed pregnancy

Fundamental Problems:

  1. Legal ownership unclear: Who controls cell phone data?
  2. Informed consent impossible: Comprehensive â‰ comprehensible
  3. True anonymization impossible: Everyone unique in high-dimensional data
  4. No data stewards: Interconnected data (social networks) affects non-participants
  5. “Tyranny of minority”: Few volunteers expose information about many

Access Provision Methods

Statistical Disclosure Control (SDC):

  • Tabular data: Safe but limited analytical value
  • Public use files: Strip PII, top-code, swap records (vulnerable to mosaic effect)
  • Licensing: Contractual agreements with monitoring
  • Differential privacy: Add calibrated noise (Îµ parameter controls privacy/utility)
  • Synthetic data: Generate fake data preserving statistical properties

Research Data Centers (RDC): Based on “Five Safes” framework:

  1. Safe projects: Legitimate research purposes
  2. Safe people: Qualified, trained researchers
  3. Safe data: Assessed reidentification risk
  4. Safe settings: Physical/technical access controls
  5. Safe outputs: Reviewed results before publication

Federal Statistical Research Data Centers

Extremely Restrictive Process:

  • 6+ months approval time
  • Special Sworn Status (security clearance + fingerprinting)
  • US citizens/3-year residents only
  • Detailed proposals with specific models pre-specified
  • No “browsing” or exploratory analysis allowed
  • All outputs must be pre-approved
  • High fees for non-affiliated researchers

Alternative: Administrative Data Research Facility (ADRF)

  • Cloud-based, virtual data facility
  • Implements “Five Safes” with modern technology
  • 450+ researchers served in 3 years
  • Government Innovation Award winner

Non-Tabular Data Privacy

Text Anonymization:

  • Simple: Remove PII via regex/dictionaries
  • Advanced: ML approaches balancing confidentiality and utility

Image/Video Anonymization:

  • Face detection and blurring/removal
  • Extending to clothing, body parts
  • Automatic anonymization while preserving data distribution

Examples: Medical records, crime notes, body camera footage, security cameras

Modern Legal and Ethical Framework

Regulatory Responses:

  • GDPR (Europe, 2018): Purpose limitation, data minimization
  • CCPA (California, 2020): Consumer data rights, processing time limits
  • Still rely on traditional notice/consent (which has failed)

Ohm’s Five Principles for Reform:

  1. Rules reflect varying risk levels across datasets
  2. Rethink traditional PII definitions
  3. Regulate “walls” between datasets
  4. Remind analysts they work with people
  5. Keep ethics discussion open and evolving

Key Insight: Move from individual consent to contextual integrity and harm assessment

Future Directions

Toward New Models: - Professionalized data curation: From artisan activity to research infrastructure - “Analysis to data”: Bring tools to secure environments vs. moving data to analysts
- Verification servers: Assess quality of inferences without exposing raw data

Integrated Approach Needed:

Combine insights from statistics, cybersecurity, cryptography, and practical applications to develop robust frameworks for human data protection

Integration: Ethical Data Science Framework

The Interconnected Challenges

How The Three Areas Connect:

Data Quality â†’ Bias: Poor data quality amplifies bias

  • Systematic errors create unfair outcomes across groups
  • Missing data mechanisms may be non-ignorable and group-specific

Bias â†’ Privacy: Bias mitigation requires detailed demographic analysis

  • Need protected characteristics to measure fairness
  • Granular analysis increases reidentification risk

Privacy â†’ Quality: Privacy protection can degrade data quality

  • Anonymization techniques reduce analytical utility
  • Restricted access limits quality validation

Best Practices for Ethical Data Science

1. Systematic Error Assessment

  • Map data generating processes and potential error sources
  • Distinguish variable vs. systematic errors
  • Consider aggregation strategies to mitigate variable errors

2. Comprehensive Bias Auditing - Start with social goals, then select appropriate metrics

  • Compare ML systems to existing (non-ML) baselines
  • Create explicit performance-equity trade-off menus

3. Privacy-Preserving Analysis

  • Apply “Five Safes” framework systematically
  • Consider differential privacy for aggregate outputs
  • Use secure research environments for sensitive analysis

Case Study Integration: Criminal Justice

Data Quality Issues:

  • Row errors: Over-policing creates biased samples
  • Label bias: Arrests â‰ crimes committed
  • Systematic errors: Vary by jurisdiction, time period

Bias and Fairness:

  • Punitive application: Focus on false positive rates
  • Historical bias: Training data reflects past discrimination
  • Competing metrics: Cannot optimize all fairness measures simultaneously

Privacy Concerns:

  • High harm potential: Criminal records are sensitive
  • Network effects: Social connections create indirect exposure
  • Long-term consequences: Digital records persist indefinitely

Recommended Workflow

1. Pre-Analysis Phase

  • Document data provenance and known quality issues
  • Identify protected groups and potential bias sources
  • Assess privacy risks and harm potential
  • Select appropriate access and analysis environment

2. Analysis Phase

  • Implement systematic error detection and mitigation
  • Measure bias across multiple fairness metrics
  • Use privacy-preserving techniques for sensitive analyses
  • Maintain detailed documentation of all decisions

3. Post-Analysis Phase

  • Validate results through replication where possible
  • Assess real-world impact through controlled deployment
  • Monitor for degradation in performance or fairness over time
  • Update methods as new challenges emerge

Key Takeaways

1. No Perfect Solutions

  • All three areas involve fundamental trade-offs
  • Perfect accuracy, complete fairness, and absolute privacy are mutually incompatible
  • Strive for transparency about limitations and trade-offs

2. Context Matters

  • Appropriate approaches depend on stakes, population, and intended use
  • Social and policy goals must drive technical choices
  • Ongoing stakeholder engagement is essential

3. Evolving Field

  • New data types and methods create new challenges
  • Regulatory landscape is rapidly changing
  • Continuous learning and adaptation required

Discussion Questions

For Class Discussion:

  1. Google Flu Trends failed spectacularly. What does this teach us about the relationship between data quality and algorithmic bias?

  2. You’re building a model for college admissions. How do you balance fairness across different demographic groups while maintaining privacy of applicants?

  3. A city wants to use ML for restaurant health inspections. What are the key quality, bias, and privacy considerations? How might they conflict?

  4. Consider the “terrorist detector” example. If we can’t achieve acceptable positive predictive value for rare events, how should this influence policy decisions about surveillance systems?

  5. The GDPR requires “right to explanation” for automated decisions. How does this interact with privacy protection in ML systems?
