Ten Simple Rules for AI-Assisted Coding in Science

Overview

The paper

Ten Simple Rules for AI-Assisted Coding in Science

Bridgeford, Campbell, Chen, Lin, Ritz, Vandekerckhove & Poldrack (2025)

AI coding tools have transformed scientific software development — but they introduce critical questions about code quality and scientific validity.

These ten rules provide a practical framework for balancing AI capabilities with methodological rigor.

The AI coding landscape

  • GitHub Copilot, ChatGPT, Claude — democratized access to programming assistance
  • Agentic tools: Claude Code, Cursor, Aider — entire coding workflows
  • Measurable productivity gains in some studies (Peng et al., 2023)

But the evidence is contested:

  • RCTs with experienced developers found AI tools slowed completion times (Becker & Rush, 2025)
  • ~200M lines of code analyzed: increases in copy-paste, decreases in refactoring (Harding et al., 2024)

Why scientific coding is different

Scientific code is not just a means to an end — it embodies:

  • Scientific reasoning
  • Methodological decisions
  • Domain expertise

Validity, reproducibility, and interpretability of scientific software directly impact research integrity and the reliability of findings.

Four key themes

Theme 1: Preparation & Understanding

Rule 1: Gather domain knowledge before implementation

Know your problem space before writing code.

  • Understand data shapes, missing data patterns, field-specific libraries
  • Identify existing implementations that could serve as models
  • Use AI to research domain standards before diving into coding

This reconnaissance phase prevents:

  • Reinventing wheels
  • Violating field conventions

Share your current understanding level with the AI and iteratively build context through targeted questions.

Rule 1 in practice

Rule 2: Distinguish problem framing from coding

Two distinct activities:

Programmatic problem framing — problem solving: understanding the domain, decomposing complex problems, finding abstractions, designing algorithms, making architectural decisions

Coding — mechanical translation of concepts into executable syntax

AI tools excel at coding tasks but require human guidance for problem framing decisions involving domain expertise, methodological choices, and scientific reasoning.

You can’t effectively guide or review what you don’t understand.

The “vibe coding” trap

Without foundational preparation, you risk “vibe coding”:

  • Accepting AI-generated code you cannot evaluate
  • Cannot debug or maintain the code
  • Unable to distinguish between elegant solutions and convoluted workarounds

Establish fluency in at least one programming language and fundamental concepts before leveraging AI assistance.

Rule 3: Choose appropriate AI interaction models

Tool Type Best For Trade-offs
Conversational (ChatGPT, Claude) Architecture design, complex debugging, learning concepts Deep reasoning but requires manual code transfer; loses context between sessions
IDE Assistant (Copilot, IntelliSense) Code completion, refactoring, maintaining flow Seamless integration but limited architectural reasoning
Autonomous Agent (Cursor, Claude Code, Aider) Rapid prototyping, multi-file changes, large refactoring High-speed but risks code divergence; requires careful monitoring

Match the tool to the task — no single paradigm fits all development phases.

Theme 2: Context Engineering & Interaction

The context challenge

Most AI systems are stateless — they forget previous interactions.

Even within sessions, context rot can degrade performance:

Context rot is the phenomenon where an LLM’s attention to mid-document details degrades as the conversation or input grows longer — even when the information is still technically within the model’s context window. The model “forgets” or loses track of earlier instructions and constraints, leading to inconsistent or lower-quality outputs.

Two critical challenges:

  • Ensuring the AI has all necessary information to generate appropriate code
  • Recognizing when a conversation has become too polluted to be productive

Rule 4: Start by thinking through a potential solution

Before engaging AI, articulate:

  • What are the inputs and expected outputs?
  • What are the key constraints and edge cases?
  • What does success look like?

Edge cases are unusual or extreme inputs that sit at the boundaries of what your code is expected to handle — e.g., empty datasets, negative values where only positive are expected, extremely large inputs, or special characters in text fields. These are where bugs most often hide.

When you provide problem context along with architectural details, the AI generates code that fits naturally into your project rather than creating isolated solutions.

This transforms the AI from a code generator into an architecture-aware development partner.

Rule 5: Manage context strategically

Context is everything in AI-assisted coding.

  • Provide all necessary information upfront: documentation, references, structured project files
  • Don’t assume the AI retains perfect context across long conversations
  • Explicitly restate critical requirements when interactions get complex
  • Keep track of context and clear/compact when approaching limits

Use externally-managed context files to maintain consistency:

  • Memory files — project-specific information: architectural decisions, standards, lessons learned
  • Constitution files — non-negotiable principles governing AI behavior
  • Problem solving files — track problems and model progress

Theme 3: Testing & Validation

Why testing matters even more with AI

AI can accelerate code generation, but it cannot be trusted to:

  • Ensure correctness
  • Handle edge cases appropriately
  • Validate that code meets scientific standards

Testing becomes the critical safeguard between AI-generated code and scientific validity.

Rule 6: Implement test-driven development with AI

Test-Driven Development (TDD) is a software development practice where you write tests first — before writing any implementation code.

The TDD cycle:

  1. Red — write a failing test that defines desired behavior
  2. Green — write the minimum code to make it pass
  3. Refactor — clean up the code while keeping tests green

Frame test requirements as behavioral specifications before requesting implementation code.

  • Tell the AI what success looks like through concrete test cases
  • Force yourself to articulate edge cases, expected inputs/outputs, and failure modes
  • AI responds better to specific test scenarios than vague functionality descriptions

Watch out for:

  • Models modifying tests to pass without solving the problem
  • Placeholder data or mock implementations that satisfy test structure without validating logic
  • “Paper tests” — seemingly passing tests that mask broken or incomplete logic

Rule 7: Leverage AI for test planning and refinement

AI is exceptionally good at identifying edge cases you might miss:

  • Boundary conditions (e.g., zero-length arrays, max integer values)
  • Type validation (e.g., passing strings where numbers are expected)
  • Error handling (e.g., network failures, missing files)
  • Numerical stability (e.g., floating-point precision, division by near-zero)

Ask AI to:

  • Generate tests for malformed inputs, extreme values, unexpected conditions
  • Review existing tests and identify gaps in coverage
  • Implement parameterized tests, fixtures, and mocking
  • Generate CI/CD boilerplate (GitHub Actions, pre-commit hooks)

Theme 4: Code Quality & Assurance

The quality challenge

AI-generated code can confidently produce outputs that:

  • Pass tests but violate domain conventions
  • Introduce subtle bugs
  • Solve problems in scientifically inappropriate ways

Human oversight remains essential — even with comprehensive tests.

Rule 8: Monitor progress and know when to restart

Actively monitor what the AI is doing:

  • Is it changing things you didn’t want changed?
  • Is it ignoring the changes you requested?
  • Is it introducing new problems while fixing old ones?

When conversations become too convoluted:

  1. Review prompt history to identify what went wrong
  2. Were requirements unclear? Conflicting constraints? Missing details?
  3. Start fresh with lessons learned — often more efficient than debugging polluted context

Use version control: commit before major changes so you can easily revert if the model goes astray.

Rule 9: Critically review generated code

Be skeptical about AI’s claims of success.

  • Models tend to claim success even when they haven’t solved the problem
  • Always test the solution independently
  • Read and understand the code to ensure it aligns with your domain expectations

Compare generated solutions against your anticipated approach from Rule 4:

  • Does it solve the problem the way that makes sense for your domain?
  • Does it match your prior expectation of how the solution should work?

Rule 10: Refine code incrementally with focused objectives

Resist the temptation to ask AI to “improve my codebase.”

Instead, be explicit about what to improve:

  • Performance optimization
  • Code readability
  • Error handling
  • Modularity
  • Adherence to design patterns

AI excels at identifying opportunities for:

  • Extracting repeated code into reusable functions
  • Detecting poor patterns: deeply nested conditionals, overly long functions, tight coupling, inconsistent naming

Always test after each incremental change — “improvements” can break working code.

Ethical Considerations

Scientific accountability

When code that generates published results is partly AI-generated, who bears responsibility?

The answer must be unequivocal: the scientist.

  • AI tools are instruments — the researcher remains fully accountable
  • “AI wrote it” is not a valid defense for flawed methodology
  • Transparency about AI usage does not diminish this responsibility

Broader ethical concerns

  • Environmental costs: training and running LLMs consumes enormous energy
  • Intellectual property: unsettled questions about training on open-source code and ownership of AI-generated code
  • Reproducibility: documentation of AI tool usage, prompting approaches, and validation procedures

Using AI coding tools involves participating in systems with significant unresolved ethical dimensions.

Guardrails for autonomous agents

Autonomous agents can make extensive changes with minimal human intervention.

Recommended safeguards:

  • Use containerized/sandboxed environments for agent-driven development
  • Commit working code before allowing agent changes (easy rollback)
  • Configure agents with explicit constraints about what they can modify
  • Maintain active monitoring rather than unsupervised operation

Sample constitution file for an autonomous agent

A constitution file sets non-negotiable rules the AI must follow throughout a project. Here is an example:

# PROJECT CONSTITUTION — Climate Data Analysis Pipeline

## Non-Negotiable Rules
1. NEVER modify raw data files in /data/raw/ — all transformations
   write to /data/processed/
2. All statistical tests must use two-tailed tests unless explicitly
   approved in the issue tracker
3. Missing values: use listwise deletion only; no imputation without
   PI approval
4. Random seeds must be set and documented for ALL stochastic operations

## Coding Standards
- Python 3.11+, type hints required on all public functions
- Pandas operations must use .copy() to avoid SettingWithCopyWarning
- No hardcoded file paths — use pathlib and config.yaml

## Forbidden Actions
- Do NOT install new packages without adding to requirements.txt
- Do NOT push directly to main — all changes via pull request
- Do NOT modify test fixtures without updating the changelog

## Scientific Constraints
- Temperature data must remain in Kelvin internally; convert only
  at output
- Confidence intervals: report 95% CI unless otherwise specified
- All plots must include axis labels, units, and source attribution

Key Concepts

Essential vocabulary

LLMs — neural networks trained on vast text corpora; generate text by predicting token sequences

Context windows — maximum tokens an LLM can consider; exceeding limits causes information loss

Context rot — degradation of attention to mid-document details as context grows, even when information is still within the context window

In-context learning — models adapt behavior based on examples and instructions in the current conversation without permanent changes to the model

Prompting — techniques for structuring inputs to elicit desired outputs, including clear specifications, examples, and formatting

Test-driven development (TDD) — writing tests before implementation to specify expected behavior and validate correctness

Externally-managed context files

These files transform AI interactions into consistent, goal-directed collaboration by providing persistent context that AI systems lack natively.

The bottom line

Even when following these rules, flawless start-to-finish interactions are the exception. The value lies in providing a framework to diagnose what went wrong and iterate more effectively.
